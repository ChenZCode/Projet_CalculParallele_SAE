\section{Introduction}

Ce projet étudie la parallélisation d'une simulation simplifiée de propagation
d'une épidémie. La population est représentée par une grille carrée de
$N \times N$ cellules. Chaque cellule correspond à une personne dont l'état
vaut 0 si elle est saine, 1 si elle est infectée et 2 si elle est guérie.

À chaque itération, une personne saine devient infectée si au moins deux de ses
quatre voisins orthogonaux sont infectés. Une personne infectée devient guérie
à l'itération suivante, tandis qu'une personne guérie conserve son état. Trois
personnes infectées sont placées au centre de la grille au début de la
simulation.

L'objectif est d'abord de comprendre les dépendances du programme séquentiel,
puis de proposer deux versions parallèles :
\begin{itemize}
	\item une version OpenMP utilisant plusieurs threads sur une mémoire partagée ;
	\item une version Open MPI utilisant plusieurs processus et des échanges de
	messages.
\end{itemize}

\section{Version séquentielle}

\subsection{Représentation des données}

La grille est stockée dans un tableau à une dimension. La macro
\texttt{IDX(i,j)} transforme les coordonnées $(i,j)$ en un indice linéaire :

\begin{lstlisting}
#define IDX(i,j) ((i)*N + (j))
\end{lstlisting}

Deux tableaux de même taille sont utilisés. Le tableau \texttt{A} contient
l'état courant de la population et le tableau \texttt{B} reçoit l'état de
l'itération suivante. Ce double stockage évite qu'une cellule déjà mise à jour
modifie le calcul de ses voisines pendant la même itération.

\subsection{Boucle principale}

La simulation répète le calcul pendant \texttt{ITER} itérations. Seules les
cellules internes sont mises à jour, car les bords ne possèdent pas quatre
voisins.

\begin{lstlisting}
for (int it = 0; it < ITER; it++) {
    for (int i = 1; i < N - 1; i++) {
        for (int j = 1; j < N - 1; j++) {
            /* Lecture dans A et ecriture dans B */
        }
    }

    int *tmp = A;
    A = B;
    B = tmp;
}
\end{lstlisting}

Pour une cellule donnée, le programme lit uniquement \texttt{A[IDX(i,j)]} et
ses quatre voisins. Il écrit le résultat dans une position distincte de
\texttt{B}. Les calculs des cellules d'une même itération sont donc
indépendants. Cette propriété rend possible la parallélisation de la boucle
portant sur les lignes.

\subsection{Résultat de référence}

Après la dernière itération, le programme compte les personnes saines,
infectées et guéries. Ce comptage sert à vérifier les versions parallèles :
pour les mêmes valeurs de $N$ et de \texttt{ITER}, elles doivent produire les
mêmes compteurs que la version séquentielle.

\section{Parallélisation avec OpenMP}

\subsection{Principe}

OpenMP repose sur une mémoire partagée. Tous les threads accèdent donc aux
mêmes tableaux \texttt{A} et \texttt{B}. La parallélisation est ajoutée avec
des directives \texttt{\#pragma omp parallel for} devant les boucles dont les
itérations sont indépendantes.

\subsection{Boucles parallélisées}

Chaque case des tableaux est écrite une seule fois. Leur initialisation peut
donc être répartie entre les threads :

\begin{lstlisting}
#pragma omp parallel for
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        A[IDX(i,j)] = 0;
        B[IDX(i,j)] = 0;
    }
}
\end{lstlisting}

La boucle principale est parallélisée sur les lignes. À chaque itération,
\texttt{A} reste en lecture seule pendant que les threads écrivent dans des
lignes différentes de \texttt{B}.

\begin{lstlisting}
for (int it = 0; it < ITER; it++) {
    #pragma omp parallel for
    for (int i = 1; i < N - 1; i++) {
        for (int j = 1; j < N - 1; j++) {
            /* Regle de propagation */
        }
    }

    int *tmp = A;
    A = B;
    B = tmp;
}
\end{lstlisting}

Enfin, le comptage utilise une réduction. Chaque thread calcule des compteurs
locaux, puis OpenMP les additionne dans \texttt{c0}, \texttt{c1} et
\texttt{c2}. Sans réduction, plusieurs threads pourraient modifier
simultanément les mêmes variables.

\begin{lstlisting}
#pragma omp parallel for reduction(+:c0,c1,c2)
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        if (A[IDX(i,j)] == 0) c0++;
        else if (A[IDX(i,j)] == 1) c1++;
        else c2++;
    }
}
\end{lstlisting}

\subsection{Compilation et exécution}

\begin{lstlisting}[numbers=none]
gcc -fopenmp epidemie_omp.c -o epidemie_omp
./epidemie_omp
\end{lstlisting}

Le nombre de threads est fixé dans le programme :

\begin{lstlisting}[numbers=none]
omp_set_num_threads(8);
\end{lstlisting}

Le temps écoulé est mesuré avec \texttt{omp\_get\_wtime}.

\subsection{Mesures obtenues}

L'accélération, ou \textit{speedup}, est calculée par
$S(p)=T(1)/T(p)$, où $p$ est le nombre de threads.

Le premier test utilise $N=2000$, soit 4 000 000 de cellules, et
\texttt{ITER=500}.

\begin{center}
\begin{tabular}{|c|c|c|}
	\hline
	\textbf{Threads} & \textbf{Temps} & \textbf{Accélération} \\ \hline
	1 & 9,333 s & 1,000 \\ \hline
	2 & 4,998 s & 1,867 \\ \hline
	4 & 6,076 s & 1,536 \\ \hline
	8 & 3,003 s & 3,108 \\ \hline
\end{tabular}
\end{center}

Le second test utilise $N=8312$, soit 69 089 344 cellules, et
\texttt{ITER=500}. Cette taille est proche de la population française et sert
ici uniquement d'ordre de grandeur.

\begin{center}
\begin{tabular}{|c|c|c|}
	\hline
	\textbf{Threads} & \textbf{Temps} & \textbf{Accélération} \\ \hline
	1 & 2 min 33 s & 1,00 \\ \hline
	2 & 1 min 30 s & 1,70 \\ \hline
	4 & 1 min 17 s & 1,99 \\ \hline
	8 & 48 s & 3,19 \\ \hline
\end{tabular}
\end{center}

Dans les deux tests, huit threads donnent le meilleur temps. L'accélération
n'est cependant pas proportionnelle au nombre de threads. Le résultat à quatre
threads sur la petite grille est même moins bon qu'à deux threads. Cela peut
s'expliquer par le coût de création et de synchronisation des threads, par le
partage de la bande passante mémoire et par la variabilité des mesures.

\section{Parallélisation avec Open MPI}

\subsection{Mémoire distribuée et initialisation}

MPI (\textit{Message Passing Interface}) permet de programmer plusieurs
processus qui possèdent chacun leur propre mémoire. Open MPI est
l'implémentation utilisée pour compiler et lancer le programme. Tous les
processus exécutent \texttt{epidemie\_mpi.c}, mais chacun travaille sur une
partie différente de la grille.

\begin{lstlisting}
MPI_Init(&argc, &argv);
MPI_Comm_rank(MPI_COMM_WORLD, &rang);
MPI_Comm_size(MPI_COMM_WORLD, &nbproc);
\end{lstlisting}

\texttt{MPI\_Init} démarre l'environnement MPI.
\texttt{MPI\_Comm\_rank} fournit le numéro du processus courant et
\texttt{MPI\_Comm\_size} le nombre total de processus. Le communicateur
\texttt{MPI\_COMM\_WORLD} contient tous les processus lancés. L'environnement
est terminé par \texttt{MPI\_Finalize}.

\subsection{Découpage de la grille}

La grille est découpée horizontalement en blocs de lignes :

\begin{lstlisting}
int bloc = N / nbproc;
int debut = rang * bloc;
\end{lstlisting}

Le processus de rang \texttt{rang} traite \texttt{bloc} lignes à partir de la
ligne globale \texttt{debut}. Le programme impose que $N$ soit divisible par
le nombre de processus. Pour $N=300$ et quatre processus, chaque processus
traite 75 lignes.

\begin{center}
\begin{tabular}{|c|c|}
	\hline
	\textbf{Processus} & \textbf{Lignes globales} \\ \hline
	0 & 0 à 74 \\ \hline
	1 & 75 à 149 \\ \hline
	2 & 150 à 224 \\ \hline
	3 & 225 à 299 \\ \hline
\end{tabular}
\end{center}

Les trois personnes initialement infectées sont placées uniquement par le
processus qui possède leurs lignes globales.

\subsection{Lignes fantômes}

La première et la dernière ligne d'un bloc dépendent de cellules appartenant
aux processus voisins. Chaque processus alloue donc deux lignes
supplémentaires :

\begin{lstlisting}
int *A = malloc((size_t)(bloc + 2) * N * sizeof(int));
int *B = malloc((size_t)(bloc + 2) * N * sizeof(int));
\end{lstlisting}

\begin{itemize}
	\item la ligne locale 0 reçoit la frontière du processus précédent ;
	\item les lignes 1 à \texttt{bloc} contiennent le bloc réellement calculé ;
	\item la ligne \texttt{bloc + 1} reçoit la frontière du processus suivant.
\end{itemize}

Ces lignes fantômes donnent accès aux voisins verticaux sans partager
directement la mémoire des autres processus.

\subsection{Échange des frontières}

Avant chaque itération, les processus voisins échangent leurs lignes extrêmes
avec \texttt{MPI\_Send} et \texttt{MPI\_Recv}.

\begin{lstlisting}
MPI_Send(&A[IDX(bloc,0)], N, MPI_INT, rang + 1, 0,
         MPI_COMM_WORLD);
MPI_Recv(&A[IDX(bloc + 1,0)], N, MPI_INT, rang + 1, 1,
         MPI_COMM_WORLD, MPI_STATUS_IGNORE);
\end{lstlisting}

Les communications sont organisées en deux phases : d'abord entre les paires
0--1, 2--3, etc., puis entre les paires 1--2, 3--4, etc. Dans chaque paire,
l'ordre des envois et des réceptions dépend de la parité du rang. Cette
organisation évite que deux processus exécutent en même temps un envoi bloquant
en attendant l'autre.

Une fois les frontières reçues, chaque processus applique la règle de
propagation à ses lignes locales. Les pointeurs \texttt{A} et \texttt{B} sont
ensuite échangés localement, comme dans la version séquentielle.

\subsection{Réduction des résultats}

Chaque processus compte les états présents dans son bloc. Une réduction
additionne ensuite les compteurs locaux sur le processus 0 :

\begin{lstlisting}
MPI_Reduce(compteurs_locaux, compteurs_globaux, 3,
           MPI_INT, MPI_SUM, 0, MPI_COMM_WORLD);
\end{lstlisting}

Une seconde réduction est utilisée pour les temps locaux. Seul le processus 0
affiche les résultats, ce qui évite plusieurs affichages identiques.

\subsection{Compilation et exécution}

\begin{lstlisting}[numbers=none]
mpicc epidemie_mpi.c -o epidemie_mpi
mpirun -np 4 ./epidemie_mpi
\end{lstlisting}

L'option \texttt{-np 4} lance quatre processus. Le découpage actuel impose que
le nombre de processus divise $N$.

\section{Analyse comparative}

\subsection{Différences entre OpenMP et MPI}

\begin{center}
\begin{tabular}{|l|c|c|}
	\hline
	\textbf{Caractéristique} & \textbf{OpenMP} & \textbf{MPI} \\ \hline
	Unité parallèle & Thread & Processus \\ \hline
	Mémoire & Partagée & Distribuée \\ \hline
	Répartition & Boucles & Blocs de lignes \\ \hline
	Communication & Variables communes & Messages \\ \hline
	Résultat global & Réduction OpenMP & \texttt{MPI\_Reduce} \\ \hline
\end{tabular}
\end{center}

OpenMP demande peu de modifications, car les threads accèdent directement à la
grille complète. MPI nécessite un découpage explicite des données, des lignes
fantômes et des communications entre processus. En contrepartie, MPI peut
fonctionner sur plusieurs machines, tandis qu'OpenMP est limité à la mémoire
partagée d'une machine.

\subsection{Validation et limites des mesures}

La première vérification porte sur les compteurs finaux. À paramètres
identiques, les trois versions doivent produire le même nombre de personnes
saines, infectées et guéries. Cette comparaison permet de détecter une erreur
de synchronisation, de réduction ou d'échange de frontière.

Les tableaux précédents permettent d'analyser OpenMP, mais ils ne permettent
pas encore de comparer directement OpenMP et MPI. Les mesures OpenMP ont été
réalisées avec 500 itérations et des grilles de taille 2000 ou 8312, tandis que
la version MPI utilise par défaut $N=300$ et 200 itérations. Une comparaison
équitable exige la même grille, le même nombre d'itérations, la même machine et
plusieurs répétitions pour chaque nombre de threads ou de processus.

La mesure du temps MPI pourrait également être améliorée. Le programme utilise
\texttt{clock()} et affiche la moyenne des temps locaux. Pour représenter la
durée réellement perçue, il serait préférable de mesurer le temps écoulé et de
retenir le temps du processus le plus lent, puisque l'exécution globale dépend
de sa terminaison.

\section{Conclusion}

Le programme séquentiel se parallélise naturellement parce que toutes les
cellules d'une itération lisent l'ancien tableau \texttt{A} et écrivent dans le
nouveau tableau \texttt{B}. OpenMP exploite cette indépendance en répartissant
les lignes entre plusieurs threads et en utilisant une réduction pour les
compteurs. Les mesures montrent un gain de temps, avec une meilleure
performance observée à huit threads, mais une accélération non linéaire.

La version MPI applique la même règle de calcul dans un modèle à mémoire
distribuée. La grille est découpée en blocs horizontaux et les processus
échangent leurs frontières avant chaque itération. Les résultats locaux sont
ensuite réunis avec \texttt{MPI\_Reduce}.

Les deux approches répondent donc au même objectif avec des modèles différents.
La suite logique du travail consiste à appliquer un protocole de mesure commun
aux trois versions afin de comparer leurs performances sans changer la taille
du problème.
